# Logging Standard

## Purpose

This standard defines how AI Film Studio produces, protects, stores, reviews, and retires application, infrastructure, security, and audit logs.

## Objectives

- Support incident detection and investigation.
- Enable reliable troubleshooting and performance analysis.
- Preserve evidence for security and compliance reviews.
- Minimize exposure of personal, confidential, and restricted data.
- Maintain consistent, searchable, and actionable telemetry.

## Scope

This standard applies to logs generated by:

- Frontend and backend applications
- APIs and WebSocket services
- Authentication and authorization systems
- Rendering and media-processing pipelines
- Databases and storage services
- CI/CD and deployment systems
- Infrastructure and network components
- Security and monitoring tools
- Administrative operations

## Logging Principles

- Log events that support security, reliability, and operational decisions.
- Do not log data without a defined purpose.
- Prefer structured, machine-readable formats.
- Use consistent field names and severity levels.
- Protect logs according to the sensitivity of their contents.
- Make timestamps accurate and comparable across systems.
- Avoid duplicate, noisy, or high-cardinality events unless justified.

## Required Event Categories

Systems should record relevant events from the following categories.

### Authentication

- Successful and failed sign-in attempts
- MFA challenges and failures
- Token issuance, refresh, revocation, and expiration
- Password or credential changes
- Account lockouts
- Suspicious authentication activity

### Authorization

- Access-denied decisions
- Role and permission changes
- Privileged access use
- Ownership changes
- Administrative impersonation or delegation

### Application and API

- Request identifiers
- Endpoint or operation name
- Response status
- Processing duration
- Rate-limit events
- Validation failures
- Dependency failures
- Significant state changes

### Data and Media Operations

- Project creation, deletion, export, and restoration
- Media upload, processing, rendering, and removal
- Sensitive data access
- Bulk operations
- Retention and deletion jobs

### Infrastructure and Deployment

- Deployment starts, completions, failures, and rollbacks
- Configuration changes
- Scaling events
- Service starts and stops
- Backup and restore operations
- Health-check failures

### Security

- Secret or key-management events
- Security-control changes
- Malware or abuse detections
- Vulnerability scanning results
- Suspicious network activity
- Audit-log access or modification attempts

## Structured Log Format

Logs should use structured JSON where supported.

Recommended fields include:

- `timestamp`
- `severity`
- `service`
- `environment`
- `event_name`
- `message`
- `request_id`
- `trace_id`
- `span_id`
- `actor_id`
- `resource_type`
- `resource_id`
- `result`
- `duration_ms`
- `error_code`
- `source_ip`
- `user_agent`
- `version`

Fields must be omitted or redacted when they would expose restricted data.

## Severity Levels

### DEBUG

Detailed diagnostic information intended for development or temporary production troubleshooting.

### INFO

Normal service activity and significant lifecycle events.

### WARN

Unexpected conditions that may require attention but do not currently prevent service operation.

### ERROR

Failures that affect a request, operation, dependency, or service function.

### CRITICAL

Severe failures, security events, or outages requiring immediate response.

Production systems should not rely on DEBUG logging by default.

## Sensitive Data

Logs must never contain:

- Passwords
- Raw access or refresh tokens
- API keys
- Private keys
- Session cookies
- Authentication headers
- Encryption secrets
- Recovery codes
- Full payment data
- Unredacted regulated identifiers
- Private media content

Minimize or redact:

- Email addresses
- IP addresses
- User-generated text
- Project names
- File names
- Support content
- Device identifiers
- Location data

Use stable internal identifiers instead of direct personal data where practical.

## Error Logging

Error logs should include enough context to diagnose the failure without exposing sensitive data.

Include:

- A stable error code
- A correlation or request identifier
- The affected operation
- The service and version
- Sanitized exception details
- Dependency status where relevant

Do not return internal stack traces to external clients.

## Correlation and Tracing

- Assign a unique request or correlation identifier at system boundaries.
- Propagate identifiers across internal services.
- Include trace and span identifiers when distributed tracing is enabled.
- Make identifiers available to support and incident responders.
- Do not use secrets or personal data as correlation identifiers.

## Timestamp Requirements

- Use UTC for persisted log timestamps.
- Use an unambiguous ISO 8601 representation.
- Synchronize system clocks through approved time sources.
- Monitor material clock drift.
- Preserve the original event time when logs are forwarded or transformed.

## Collection and Transport

- Transmit logs over encrypted channels.
- Authenticate log producers and collectors.
- Buffer safely during temporary collector outages.
- Prevent untrusted input from altering log structure.
- Monitor ingestion failures and dropped events.
- Separate production and non-production log streams.

## Storage and Access

- Store logs in approved centralized systems.
- Apply least-privilege access.
- Require strong authentication for administrative access.
- Restrict security and audit logs more tightly than routine operational logs.
- Encrypt logs in transit and at rest.
- Record administrative access to sensitive log stores.
- Protect logs from unauthorized modification or deletion.

## Retention

Retention must follow `docs/DATA_RETENTION_POLICY.md`.

Retention periods should consider:

- Operational troubleshooting needs
- Security investigation requirements
- Contractual obligations
- Legal or regulatory requirements
- Data minimization principles
- Storage cost

Logs containing personal or confidential data should have the shortest practical retention period.

## Monitoring and Alerting

Alerts should be based on actionable conditions, including:

- Elevated authentication failures
- Repeated authorization denials
- Unusual privileged activity
- Error-rate or latency increases
- Service health degradation
- Unexpected deployment failures
- Log-ingestion interruptions
- Attempts to disable or modify logging
- Abnormal data export or deletion activity

Every alert should have:

- An owner
- A severity
- A notification route
- A runbook or response procedure
- A review cadence

## Audit Logs

Audit logs must record security-relevant administrative and data-access actions.

Audit records should identify:

- Who performed the action
- What action occurred
- Which resource was affected
- When the action occurred
- The result
- The originating system or location
- Relevant request or change identifiers

Audit logs should be append-oriented and protected from alteration.

## Development and Testing

- Use synthetic or anonymized data in development logs.
- Do not copy unrestricted production logs into local environments.
- Ensure test fixtures do not contain real credentials.
- Disable temporary verbose logging after troubleshooting.
- Review logging changes as part of code review.

## Availability and Failure Handling

Logging failures must not silently undermine security monitoring.

Systems should:

- Detect collector and pipeline failures.
- Expose health metrics for logging components.
- Buffer within safe limits.
- Apply backpressure or sampling when necessary.
- Avoid blocking critical service functions unless audit requirements demand it.
- Alert when security or audit logs are lost.

## Review and Testing

Logging controls should be reviewed at least annually and after significant architectural changes.

Testing should verify:

- Required events are emitted.
- Sensitive fields are redacted.
- Correlation identifiers propagate correctly.
- Timestamps are accurate.
- Alerts trigger as expected.
- Retention and deletion rules operate correctly.
- Access controls prevent unauthorized viewing or modification.
